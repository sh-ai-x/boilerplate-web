## §1 Frame

- **goal**: Ship `npx create-boilerplate-web <folder> --type=<saas|shop|portfolio>` that targets a sub-folder of the master repo, plus three independently buildable Next.js + Supabase + Cloudflare + Toss Payment templates, with payment logic isolated to Supabase Edge Functions and prices controlled from an Admin DB layer.
- **target user**: Solo indie hacker shipping one product at a time and wanting production-grade auth/payments/security without owning the boilerplate.
- **situation**: `<unspecified>` (user deferred — "you figure it out"). Defaulting to: "they paste snippets, hand-roll Supabase RLS, ship payment forms where the client controls the amount, and re-discover the same auth bypass in every new project."

## §3 Non-goals

| # | Non-goal | Rationale | Breach-response |
|---|----------|-----------|-----------------|
| 1 | **No email/password or magic-link auth — Google OAuth only.** | v1 ships one auth path to keep RLS surface + Edge Function guard logic small; adding email/PW multiplies the password-reset / session-refresh surface area. | If reviewer asks for email/PW: open a v2 PRD (`feat(auth-email)`), do NOT graft it into this phase. |
| 2 | **No payment providers beyond Toss.** | Toss is the only MCP wired into the meta-prompt; adding Stripe/PayPal requires parallel billing-key state machines and breaks the single Edge Function contract. | If reviewer asks for Stripe: open a separate `templates/saas-stripe` repo, do NOT fork the Edge Function. |
| 3 | **No multi-tenant Supabase — each user brings their own Supabase project.** | Multi-tenancy needs row-level isolation across tenants + a project-provisioning Edge Function, which is a different product. | If reviewer asks for multi-tenancy: this becomes a separate SaaS-of-SaaS product, NOT a boilerplate. |
